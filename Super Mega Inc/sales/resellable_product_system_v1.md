# SuperMega Resellable Product System (V1)

## Review of the Current Public Lineup

The current public lineup is better than before, but it still has one structural weakness:

- it is clear enough to demo
- it is not yet detailed enough to sell repeatedly

### What is working

- the products are simpler
- the free tools are easy to understand
- the ladder from tool -> module -> OS is directionally right

### What is not working

- `Lead Finder`, `News Brief`, and `Action Board` are useful as proof tools, but too thin to be the main commercial products
- the current modules are described as features, not as repeatable packaged systems
- there is no clear template for adapting the same product to a different client data context
- there is no standard rollout model per product

## Product Rule

Public free tools are the proof layer.

Sellable products must be:

- repeatable
- data-adaptable
- owner-based
- evidence-linked
- usable within one function before full OS rollout

## Standard Product Template Structure

Every sellable SuperMega product should follow the same structure:

1. buyer
2. problem
3. required data
4. core workflow
5. outputs
6. rollout model
7. reuse pattern

## Product Template 1: Action OS

### Buyer

- founder
- general manager
- operations director
- plant manager

### Problem

Important work is buried across emails, sheets, chats, and verbal updates. Managers spend time chasing status instead of driving action.

### Required Data

- daily update sheet or form
- forwarded notes or management emails
- optional shared action tracker
- owner list and team structure

### Core Workflow

1. ingest daily updates from existing tools
2. classify each update by function, urgency, and owner
3. convert updates into action items with due windows
4. group actions into a manager board and a director summary
5. track open, overdue, blocked, and closed items

### Outputs

- live action board
- director brief
- owner queue
- blocker list
- overdue action report

### Rollout Model

- week 1: connect one update source and owner list
- week 2: generate daily action board and review discipline
- week 3+: add write-back and manager follow-up rhythm

### Reuse Pattern

Only the input connectors, owner mapping, and issue taxonomy change per client.
The core action engine stays the same.

## Product Template 2: Supplier Watch

### Buyer

- procurement manager
- supply chain lead
- operations director

### Problem

Supplier risk appears too late. Delay, payment, documentation, and customs issues are scattered across inboxes and spreadsheets.

### Required Data

- supplier emails
- PO / ETA sheet
- payment status sheet
- shipment or customs notes
- supplier master list

### Core Workflow

1. ingest supplier communications and ETA records
2. detect payment, delay, customs, and document risk
3. score the risk per thread or shipment
4. assign owner and due date
5. publish a supplier risk queue and escalation view

### Outputs

- supplier risk board
- escalation queue
- owner follow-up list
- supplier summary by risk type
- reply draft support

### Rollout Model

- week 1: connect one mailbox and one ETA tracker
- week 2: run risk scoring and owner assignment
- week 3+: add escalation rules and review rhythm

### Reuse Pattern

The reusable core is the risk engine.
Per client, swap:

- supplier list
- shipment fields
- payment rules
- escalation thresholds

## Product Template 3: Quality Closeout

### Buyer

- quality manager
- QA lead
- plant manager

### Problem

Quality issues are logged inconsistently and CAPA follow-up breaks down after the first report.

### Required Data

- incident log
- quality complaint emails
- inspection evidence or file links
- supplier or product master
- owner list

### Core Workflow

1. capture issue from email, form, or sheet
2. classify severity and issue type
3. generate containment, RCA, corrective action, and verification steps
4. assign owners and target dates
5. track open, overdue, and closed CAPA items

### Outputs

- incident register
- CAPA chain
- closure checklist
- overdue CAPA list
- supplier or defect trend view

### Rollout Model

- week 1: connect incident intake and owner structure
- week 2: run incident-to-CAPA workflow
- week 3+: add trend views and closure verification discipline

### Reuse Pattern

The CAPA framework is stable across clients.
Per client, swap:

- defect taxonomy
- severity rules
- evidence standards
- role ownership map

## Product Template 4: Cash Watch

### Buyer

- finance manager
- commercial controller
- managing director

### Problem

Overdue invoices and collection risk are visible too late and follow-up discipline is weak.

### Required Data

- invoice register
- cash book or receipts sheet
- payment follow-up emails
- customer master
- collector / owner mapping

### Core Workflow

1. ingest invoices, cash records, and payment messages
2. reconcile what is open, partially paid, or overdue
3. rank accounts by urgency and value
4. assign collection follow-up to owners
5. publish a weekly cash-control view

### Outputs

- overdue queue
- collection priority list
- promised-payment tracker
- customer follow-up drafts
- management cash-control summary

### Rollout Model

- week 1: connect invoice and cash sources
- week 2: run aging and priority logic
- week 3+: add owner workflow and weekly reporting

### Reuse Pattern

The reusable core is the receivables control engine.
Per client, swap:

- invoice schema
- aging rules
- customer segments
- owner and escalation rules

## Product Template 5: Sales Signal

### Buyer

- sales manager
- commercial director
- founder

### Problem

Demand shifts, channel signals, and pricing changes are seen too late and not turned into follow-up actions.

### Required Data

- sales update sheet
- distributor or channel notes
- market headlines
- optional pricing sheet
- customer/channel master

### Core Workflow

1. ingest sales notes, channel updates, and market signals
2. detect changes in demand, pricing, mix, or channel risk
3. turn signals into actions for sales and procurement
4. publish one short sales signal brief
5. track follow-up by owner

### Outputs

- sales signal brief
- demand-shift watchlist
- follow-up queue
- pricing risk notes
- weekly commercial summary

### Rollout Model

- week 1: connect channel update source and signal tags
- week 2: generate brief and owner queue
- week 3+: connect to procurement and inventory planning

### Reuse Pattern

The signal engine is reusable.
Per client, swap:

- channel structure
- product categories
- price assumptions
- commercial review rhythm

## Product Template 6: SuperMega OS

### Buyer

- founder
- CEO
- COO
- director team

### Problem

The company runs through scattered files, inboxes, and people memory. There is no single operating layer for action, approvals, and follow-up.

### Required Data

- one or more module inputs
- owner and role model
- operating calendar
- approval rules
- evidence links or source records

### Core Workflow

1. normalize signals across functions into one shared action model
2. route actions to role-specific queues
3. manage approvals, escalations, and blocked items
4. generate director and founder command views
5. keep audit trail and evidence trace for every important action

### Outputs

- role dashboards
- founder command view
- approval queue
- blocker and exception queue
- operating scorecards

### Rollout Model

- phase 1: deploy one function template
- phase 2: connect two or three templates into one action layer
- phase 3: add approvals, scorecards, and cross-function governance

### Reuse Pattern

`SuperMega OS` is not rebuilt from zero each time.
It is assembled from:

- shared connector templates
- shared entity model
- shared action schema
- shared role dashboard system
- client-specific mappings and rules

## Commercial Recommendation

Use this structure:

- free proof layer:
  - Lead Finder
  - News Brief
  - Action Board
- sellable workflow layer:
  - Action OS
  - Supplier Watch
  - Quality Closeout
  - Cash Watch
  - Sales Signal
- flagship layer:
  - SuperMega OS

## What To Put On The Website

Do not sell the free tools as the main business.

Show them as:

- proof
- self-serve test
- lead qualification

Sell the workflow templates as the actual products.

## Implementation Rule For Reuse

Every product should ship with the same reusable package:

- connector template
- field mapping template
- owner mapping template
- rules file
- prompt profile
- dashboard template
- rollout SOP

That is what makes the products portable across clients.
